### Design Patterns and Principles
* **Exercise 1: Singleton Pattern** - Restricts a class to a single instance (e.g., a central `Logger`) and provides a global point of access.
* **Exercise 2: Factory Method Pattern** - Defines an interface for creating objects but lets subclasses decide which class to instantiate (e.g., dynamic creation of Word, PDF, or Excel documents).
* **Exercise 3: Builder Pattern** - Separates the construction of a complex object from its representation, allowing step-by-step creation of different configurations (e.g., custom `Computer` systems).
* **Exercise 4: Adapter Pattern** - Allows incompatible interfaces to work together by wrapping a class around a gateway (e.g., unifying different payment gateways like PayPal and Razorpay under a common `PaymentProcessor` interface).
* **Exercise 5: Decorator Pattern** - Dynamically adds new behavior to an object at runtime without altering its structure (e.g., wrapping an email notifier to also send SMS and Slack notifications).
* **Exercise 6: Proxy Pattern** - Provides a placeholder or surrogate for another object to control access, manage lazy loading, or cache requests (e.g., delaying expensive loading of `RealImage` until displayed).
* **Exercise 7: Observer Pattern** - Establishes a one-to-many dependency relationship where changes in one object notify all dependents (e.g., `StockMarket` notifying registered mobile and web apps).
* **Exercise 8: Strategy Pattern** - Defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime (e.g., switching between credit card and PayPal payment strategies).
* **Exercise 9: Command Pattern** - Encapsulates a request as an object, thereby letting you parameterize clients with different requests (e.g., remote control buttons executing light ON/OFF commands).
* **Exercise 10: MVC Pattern** - Separates application logic, presentation, and user input to achieve clean separation of concerns (e.g., `Student` model, `StudentView`, and `StudentController`).
* **Exercise 11: Dependency Injection** - Supplies dependencies (like repositories) to a class from the outside rather than letting the class instantiate them directly, enabling loose coupling and unit testing.
